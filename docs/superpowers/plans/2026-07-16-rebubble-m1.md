# Rebubble M1 (Core Messaging) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship M1 of **Rebubble** — a native Kotlin + Jetpack Compose Android client for the unmodified BlueBubbles macOS server: onboarding (QR + manual), conversation list, chat view, send/receive text + attachments, and never-miss-a-message notifications (FCM + reconciliation sync).

**Architecture:** Offline-first. Room is the single source of truth; UI reads only Room Flows. Three inbound paths (FCM push, socket.io while foregrounded, REST reconciliation sync) converge on one idempotent `MessageIngestor` keyed on message GUID inside a single Room transaction. Sends go through an outbox with temp GUIDs, WorkManager retries, and visible failure states.

**Tech Stack:** Kotlin, Jetpack Compose (Material 3, dynamic color), Room, Retrofit/OkHttp + kotlinx.serialization, socket.io-client (java), WorkManager, Coil, Hilt, Firebase Messaging (runtime-initialized from the user's server-supplied config), DataStore + EncryptedSharedPreferences, JUnit/Robolectric/Turbine/MockWebServer (unit + integration), Maestro (UI flows), GitHub Actions CI.

## Context

The official BlueBubbles Android app is a cross-platform Flutter codebase with chronic UI and reliability problems (upstream issues #2766, #2972). The macOS server is the solid part and exposes a REST + socket.io + FCM contract. Approved design: build a clean native client — *"Keep your server, replace the app."* Public open-source (Apache-2.0), Material 3-first with its own identity (iOS elements only as seasoning), performance as a hard requirement. This plan covers **M1 only**; the schema deliberately stores M2 fields (reactions, threads, edits) so M2 needs no migration.

User decisions locked in: name **Rebubble**, applicationId **`app.rebubble.android`**, package root **`app.rebubble`**; distribution via **GitHub Releases APK** (no Play/F-Droid work in M1); push strategy **FCM + reconciliation fallback**.

## Verified API Contract (extracted from upstream source — cite fixtures against these files)

Ground truth: `bluebubbles-server/packages/server/src/server/api/http/api/v1/httpRoutes.ts`, `.../api/serializers/{Message,Chat,Handle,Attachment}Serializer.ts`, `.../services/fcmService/index.ts`, `.../api/http/index.ts` (socket auth), `.../api/v1/middleware/authMiddleware.ts`.

- **Envelope:** `{status, message, data, metadata?, error?:{type,message}}`. Paginated metadata `{offset,limit,total,count}`. Errors: 400 "Validation Error", 401/403 "Authentication Error", 404 "Database Error", 500 "Server Error"/"iMessage Error", 504 timeout.
- **Endpoints used in M1:**
  - `GET /api/v1/ping`; `GET /api/v1/server/info` → `{server_version, os_version, private_api:bool, helper_connected:bool, ...}`
  - `POST /api/v1/chat/query` body `{with:["lastMessage","participants"], sort:"lastmessage", offset, limit≤1000}`
  - `GET /api/v1/chat/:guid/message?with=attachment,message.attributedBody,message.messageSummaryInfo&sort=DESC&before=<ms>&limit=100` (history backfill)
  - `POST /api/v1/message/query` body `{with:["chats","attachments","handles"], where:[{statement:"message.ROWID > :rowid", args:{rowid}}], sort:"ASC", limit≤1000, offset}` (reconciliation)
  - `POST /api/v1/message/text` body `{chatGuid, tempGuid, message, method:"apple-script"|"private-api"}` → sent MessageDto with `tempGuid` echoed. tempGuid is the server-side idempotency key.
  - `POST /api/v1/message/attachment` multipart: file field **`attachment`**, fields `chatGuid, tempGuid, name, method`
  - `GET /api/v1/attachment/:guid/download?original=true` (stream, real Content-Type)
  - `GET /api/v1/contact` / `POST /api/v1/contact/query` (names first; avatars lazy — can be huge base64)
  - `GET /api/v1/fcm/client` → google-services.json-shaped object; `POST /api/v1/fcm/device` body `{name, identifier:<fcm token>}`
- **Socket.io:** connect to server origin, query `{"guid": <password>}`, transports `[websocket, polling]`. Server-push events (raw payloads, NO envelope): `new-message`, `updated-message`, `typing-indicator`, `chat-read-status-changed`, `group-name-change`, `participant-added/removed/left`, `message-send-error`, `new-server`, `hello-world`.
- **FCM:** data-only messages `{data:{type:<same event names>, data:<JSON string of full payload>}}`, no `notification` block. Payload may be AES-encrypted (CryptoJS/EVP, password as passphrase) when server `encrypt_coms` on, and oversized payloads are silently dropped server-side → **client must never rely on FCM payload alone**; unparseable/absent payload = wake signal → expedited sync.
- **Onboarding QR payload:** JSON array `[password, serverUrl]` (password FIRST).
- **Key DTO fields:** MessageDto: `originalROWID, guid, tempGuid?, text, subject, error:int, dateCreated/dateRead/dateDelivered:ms, isFromMe, handle{address,service}, handleId, attachments[], itemType, groupTitle, groupActionType, associatedMessageGuid (strip `p:N/`/`bp:` prefixes), associatedMessageType ("love"/"-love"/...), threadOriginatorGuid, expressiveSendStyleId, dateEdited, dateRetracted, chats[]?`. ChatDto: `guid ("iMessage;-;+1555…"/"SMS;-;…"), style (43=group, 45=DM), chatIdentifier, displayName, participants[], lastMessage?`. AttachmentDto: `guid, uti, mimeType, transferName, totalBytes, height, width, hasLivePhoto`.
- **Verified:** server always serializes `text` as `universalText` (falls back to attributedBody extraction) — `MessageSerializer.ts:132`. **M1 skips attributedBody parsing entirely** (defensive log if `text==null && attributedBody!=null`).
- **Group events as messages** (render as centered system rows): itemType 1 = participant add(0)/remove(1); itemType 2 = rename (groupTitle) / name removed; itemType 3 = left(0)/photo changed(1)/photo removed(2).

## Core Design Decisions

### Room schema (v1 — includes M2 fields to avoid migration)

```kotlin
@Entity(tableName = "chats")
data class ChatEntity(
  @PrimaryKey val guid: String, val style: Int, val chatIdentifier: String?,
  val displayName: String?, val isArchived: Boolean = false,
  val lastMessageDate: Long?, val lastMessagePreview: String?,   // denormalized by ingestor
)
@Entity(tableName = "messages",
  indices = [Index("chatGuid","dateCreated"), Index("originalRowId"), Index("threadOriginatorGuid")])
data class MessageEntity(
  @PrimaryKey val guid: String,          // "temp-<8hex>" until ack swaps it
  val chatGuid: String, val originalRowId: Long?,
  val text: String?, val subject: String?, val isFromMe: Boolean, val senderAddress: String?,
  val dateCreated: Long, val dateRead: Long?, val dateDelivered: Long?, val error: Int = 0,
  val itemType: Int = 0, val groupActionType: Int = 0, val groupTitle: String?,
  val associatedMessageGuid: String?,    // stored PRE-stripped of p:N/ and bp: prefixes
  val associatedMessageType: String?, val threadOriginatorGuid: String?,
  val expressiveSendStyleId: String?, val dateEdited: Long?, val dateRetracted: Long?,
  val sendStatus: SendStatus,            // SENT | SENDING | FAILED (local-only)
)
@Entity(tableName = "attachments", indices = [Index("messageGuid")])
data class AttachmentEntity(
  @PrimaryKey val guid: String, val messageGuid: String, val uti: String?, val mimeType: String?,
  val transferName: String?, val totalBytes: Long?, val width: Int?, val height: Int?,
  val hasLivePhoto: Boolean, val localPath: String?, val downloadState: DownloadState,
)
@Entity(tableName = "handles", primaryKeys = ["address","service"])
data class HandleEntity(val address: String, val service: String)
@Entity(tableName = "chat_handles", primaryKeys = ["chatGuid","address"])
data class ChatHandleCrossRef(val chatGuid: String, val address: String)
@Entity(tableName = "contacts")
data class ContactEntity(@PrimaryKey val address: String, val displayName: String?, val avatarPath: String?)
```
Watermark + server-info cache in DataStore; password in EncryptedSharedPreferences. Reaction rows (`associatedMessageType != null`) are stored but filtered out of chat queries in M1 (rendered in M2); they still advance the watermark. Schema exported to `app/schemas/` from day one.

### MessageIngestor (the convergence point — most-tested unit)

```kotlin
enum class IngestSource { SOCKET, FCM, RECONCILE, SEND_ACK, BACKFILL }
data class IngestResult(val insertedGuids: List<String>, val updated: Int, val maxRowId: Long?)
class MessageIngestor @Inject constructor(private val db: RebubbleDatabase) {
  suspend fun ingest(dtos: List<MessageDto>, source: IngestSource): IngestResult
}
```
Per message, inside ONE `withTransaction`:
1. Row exists with same `guid` → merge-update (prefer non-null server fields; never regress `dateRead`/`dateDelivered` to null; set `sendStatus=SENT`).
2. Else if `dto.tempGuid != null` and a row exists with `guid == tempGuid` → PK swap via raw `@Query("UPDATE messages SET guid=:real, originalRowId=:rowId, sendStatus='SENT', ... WHERE guid=:temp")` (Room can't change PKs via `@Update`); re-parent attachments in the same transaction.
3. Else insert (+ upsert handles, cross-refs, attachments — attachment upsert is read-modify-write preserving `localPath`/`downloadState`).
4. Update chat `lastMessageDate`/`lastMessagePreview` if newer.

This makes the socket-echo vs REST-ack race a non-issue: both funnel into `ingest()`; whichever arrives first does the swap, the other becomes an idempotent merge. `insertedGuids` filtered to `!isFromMe` feeds notifications.

### Reconciliation: global ROWID watermark

1. `POST /chat/query` (paginated) → upsert chats/participants (catches new chats + renames).
2. Loop `POST /message/query` `ROWID > watermark`, ASC, limit 1000 → `ingest(page, RECONCILE)` → advance watermark to page max `originalROWID` **after** the transaction commits → repeat until `count < limit`.
3. Watermark init at onboarding = current max ROWID (1 message, sort DESC); history backfills lazily per chat on scroll-up via `GET /chat/:guid/message?before=`.

Triggered on: app foreground, socket reconnect, FCM wake, periodic WorkManager (15 min, network-constrained), manual refresh. Crash between ingest and watermark write only re-ingests one idempotent page.

### Outbox

```kotlin
class OutboxRepository @Inject constructor(...) {
  suspend fun sendText(chatGuid: String, text: String): String   // returns "temp-<8hex>"
  suspend fun sendAttachment(chatGuid: String, uri: Uri): String
  suspend fun retry(tempGuid: String)
}
```
Optimistic insert (`sendStatus=SENDING`, `dateCreated=now`) → unique `OneTimeWorkRequest` (tag=tempGuid, exponential backoff, network constraint). Worker: POST → on 200 pipe response through `ingest(SEND_ACK)`. IOException → `Result.retry()`; ≥3 attempts or 4xx → `sendStatus=FAILED` + failure notification. tempGuid idempotency means an ambiguous-timeout retry can't double-send. Attachment files are copied into app storage before enqueue (picker URIs expire). `method:"private-api"` when `server/info.private_api && helper_connected`, else `"apple-script"`.

### Notifications

Data-only FCM → `PushHandler`: parseable `new-message` payload → `ingest(FCM)` + notify; garbage/encrypted/absent → expedited `SyncWorker` (which notifies from `IngestResult.insertedGuids`). `MessageNotifier` builds `MessagingStyle` from Room (last ~10, `Person` from contacts, group flag from style==43), id = chat guid hash, grouped summary, suppressed when `ActiveChatTracker.current == chatGuid`. Inline reply: `RemoteInput` → `ReplyReceiver` → `OutboxRepository.sendText` → re-post notification including the pending reply. Firebase is **runtime-initialized**: `GET /fcm/client` → `FirebaseOptions` → `FirebaseApp.initializeApp(ctx, options, "rebubble")` → token → `POST /fcm/device {name: Build.MODEL, identifier: token}`. No compile-time google-services.json. FCM failure is non-fatal: banner "notifications degraded — periodic sync active".

### M1 deliberate cuts (spec-sanctioned)

attributedBody parsing (verified unnecessary); encrypted-FCM decryption (wake-signal fallback instead); tapback rendering / read-receipt POST / typing (M2 — but schema stores the fields now); blurhash placeholders (aspect-ratio boxes instead); tablet/foldable panes (phone-first; screens kept stateless so list-detail is additive).
